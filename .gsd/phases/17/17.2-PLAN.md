# Plan 17.2: Frontend Context Sharing

## Objective
Standardize how the `chamaId` is passed and accessed within the frontend application to reduce redundant API calls and prevent nested UI components from losing track of the active Chama context.

## Context
- .gsd/SPEC.md
- frontend/src/context/ChamaContext.jsx (To be created or updated)
- frontend/src/pages/chama/ChamaDashboard.jsx

## Tasks

<task type="auto">
  <name>Implement Active Chama Context Provider</name>
  <files>frontend/src/context/ChamaContext.jsx</files>
  <action>
    - Create a React Context Provider `ChamaProvider` that fetches and stores the currently active Chama details and the current user's role within that Chama.
    - It should read the `id` from the URL parameters (`useParams`).
    - Expose `activeChama`, `userRole`, and `loading` states to consumer components.
  </action>
  <verify>cat frontend/src/context/ChamaContext.jsx</verify>
  <done>ChamaContext is implemented and capable of serving active context globally.</done>
</task>

<task type="auto">
  <name>Integrate ChamaContext into Application Layout</name>
  <files>
    frontend/src/App.jsx
    frontend/src/pages/chama/ChamaDashboard.jsx
  </files>
  <action>
    - Wrap the Chama-specific routes in `App.jsx` with the `<ChamaProvider>`.
    - Refactor `ChamaDashboard.jsx` (and child components like `RoscaDashboard`) to consume `useChama()` instead of independently fetching membership and chama details.
  </action>
  <verify>grep -n "ChamaProvider" frontend/src/App.jsx</verify>
  <done>The application safely relies on the global context for the active Chama, drastically reducing redundant network requests.</done>
</task>

## Success Criteria
- [ ] Navigating deeply into Chama sub-menus (like ROSCA or Meetings) does not trigger redundant `getChama` or `getMembers` API calls.
- [ ] The user's role and permissions properly render based on the centralized Context.
